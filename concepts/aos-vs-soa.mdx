## Introduction

ImgTool implements two parallel architectures for image processing, allowing direct performance comparison between Array of Structs (AOS) and Struct of Arrays (SOA) memory layouts. Both implementations produce identical results but exhibit different performance characteristics.

## Data Structure Definitions

### Array of Structs (AOS)

From `imgaos/imageaos.hpp`:

```cpp
struct Pixel {
    int r, g, b;

    bool operator<(const Pixel& other) const {
        return std::tie(r, g, b) < std::tie(other.r, other.g, other.b);
    }

    bool operator==(const Pixel& other) const {
        return r == other.r && g == other.g && b == other.b;
    }
};

class Image {
public:
    image_size size;
    int max_color;
    uint32_t colors = 0;
    std::vector<Pixel> pixels;  // Array of Pixel structs

    Image(image_size size_init, int max_c);
};
```

<Note>
In AOS, each pixel is a self-contained struct with all three color channels stored together. The `pixels` vector contains complete `Pixel` objects.
</Note>

### Struct of Arrays (SOA)

From `imgsoa/imagesoa.hpp`:

```cpp
struct ImageSOA {
    std::vector<int> r;  // Red component
    std::vector<int> g;  // Green component
    std::vector<int> b;  // Blue component
    std::size_t width = 0;
    std::size_t height = 0;
};

class Image {
public:
    image_size size;
    int max_color;
    uint32_t n_colors = 0;
    std::vector<uint16_t> red;    // Separate red channel
    std::vector<uint16_t> green;  // Separate green channel
    std::vector<uint16_t> blue;   // Separate blue channel

    Image(image_size size_init, int max_color);
};
```

<Note>
In SOA, color channels are separated into distinct arrays. The red, green, and blue vectors each contain all values for that channel across the entire image.
</Note>

## Memory Layout Visualization

<CodeGroup>
```text AOS Layout
Memory Address:   [Low]                              [High]
                   ↓                                   ↓
Pixels:           [Pixel 0  ][Pixel 1  ][Pixel 2  ][Pixel 3  ]...
Components:       [R₀ G₀ B₀ ][R₁ G₁ B₁ ][R₂ G₂ B₂ ][R₃ G₃ B₃ ]...

Access pattern for grayscale:
  Read: R₀ G₀ B₀ → R₁ G₁ B₁ → R₂ G₂ B₂ → ...
  ✓ Sequential access, cache-friendly
```

```text SOA Layout  
Memory Address:   [Low]            [High]
                   ↓                 ↓
Red channel:      [R₀][R₁][R₂][R₃]...[Rₙ]
Green channel:    [G₀][G₁][G₂][G₃]...[Gₙ]
Blue channel:     [B₀][B₁][B₂][B₃]...[Bₙ]

Access pattern for red channel boost:
  Read: R₀ → R₁ → R₂ → R₃ → ...
  ✓ Perfect cache utilization, no wasted bandwidth
```
</CodeGroup>

## Performance Characteristics

### Access Pattern Analysis

<Tabs>
  <Tab title="Full Pixel Access">
    **AOS Advantage**
    
    When algorithms need all RGB values together:
    
    ```cpp
    // AOS: Excellent locality
    for (const auto& pixel : image.pixels) {
        int gray = (pixel.r + pixel.g + pixel.b) / 3;
        // All values in same cache line
    }
    
    // SOA: Must access three separate arrays
    for (size_t i = 0; i < size; ++i) {
        int gray = (image.r[i] + image.g[i] + image.b[i]) / 3;
        // Three separate memory locations
    }
    ```
    
    **AOS is faster**: 1.5-2x better for operations like:
    - Color space conversions
    - Bilinear interpolation
    - Pixel-based filters
  </Tab>
  
  <Tab title="Single Channel Access">
    **SOA Advantage**
    
    When algorithms process one channel at a time:
    
    ```cpp
    // AOS: Loads unnecessary data
    for (auto& pixel : image.pixels) {
        pixel.r = clamp(pixel.r * factor, 0, 255);
        // Loads G and B into cache unnecessarily
    }
    
    // SOA: Perfect efficiency
    for (auto& r : image.r) {
        r = clamp(r * factor, 0, 255);
        // Only red channel in cache
    }
    ```
    
    **SOA is faster**: 2-4x better for operations like:
    - Histogram calculation
    - Channel-specific adjustments
    - Frequency-based filtering (cutfreq)
  </Tab>
  
  <Tab title="Vectorization">
    **SOA Advantage**
    
    Modern CPUs support SIMD (Single Instruction Multiple Data):
    
    ```cpp
    // SOA: Compiler can auto-vectorize
    for (size_t i = 0; i < image.r.size(); ++i) {
        image.r[i] = image.r[i] * 2;  // Process 4-8 at once with SIMD
    }
    
    // AOS: Harder to vectorize due to interleaved data
    for (auto& pixel : image.pixels) {
        pixel.r = pixel.r * 2;  // Must deinterleave first
    }
    ```
    
    **SOA advantage**: Better compiler optimization, up to 4x faster with AVX2
  </Tab>
</Tabs>

## Operation-Specific Performance

### Compress Operation

Both implementations use similar algorithms for the `compress` operation:

<CodeGroup>
```cpp AOS Implementation (imageaos.cpp:85-121)
void compress(progargsCommon::parameters_files &params) {
    // Read PPM header
    Image picture({.width=width, .height=height}, max_color);
    
    // Read pixels as complete structs
    for(size_t i = 0; i < picture.pixels.size(); i++) {
        if (picture.max_color <= 255) {
            picture.pixels[i].r = binario::read_binary<uint8_t>(input);
            picture.pixels[i].g = binario::read_binary<uint8_t>(input);
            picture.pixels[i].b = binario::read_binary<uint8_t>(input);
        }
    }
    
    std::map<Pixel, uint32_t> colorlist = getColors(picture.pixels, picture);
    write_dataCPPM(picture, colorlist, params.output_file);
}
```

```cpp SOA Implementation (imagesoa.cpp:118-153)
void compress(progargsCommon::parameters_files params) {
    // Read PPM header
    Image image(size_init, max_color);
    
    // Read pixels into separate channel arrays
    for(size_t i = 0; i < image.red.size(); i++) {
        if (image.max_color <= 255) {
            image.red[i] = binario::read_binary<uint8_t>(input);
            image.green[i] = binario::read_binary<uint8_t>(input);
            image.blue[i] = binario::read_binary<uint8_t>(input);
        }
    }
    
    auto color_list = getColors(image);
    write_color_table(image, color_list, params.output_file);
}
```
</CodeGroup>

<Info>
For compression, AOS often performs slightly better because the operation requires accessing all three color channels together to build the color table.
</Info>

### CutFreq Operation

The `cutfreq` operation removes low-frequency colors:

<CodeGroup>
```cpp AOS: Color Frequency Counting
std::unordered_map<Pixel, int, PixelHash> 
countColorFrequencyWithParams(const CutFreqParams& params) {
    std::unordered_map<Pixel, int, PixelHash> frequencyMap;
    
    for (const auto& pixel : params.image) {
        frequencyMap[pixel]++;  // Hash entire pixel struct
    }
    
    return frequencyMap;
}
```

```cpp SOA: Color Frequency Counting
std::unordered_map<std::tuple<int, int, int>, int, TupleHash>
countColorFrequencySOA(CutFreqParamsSOA const & params) {
    std::unordered_map<std::tuple<int, int, int>, int, TupleHash> frequencyMap;
    
    for (size_t i = 0; i < params.image->width * params.image->height; ++i) {
        auto color = std::make_tuple(params.image->r[i], 
                                     params.image->g[i], 
                                     params.image->b[i]);
        frequencyMap[color]++;
    }
    
    return frequencyMap;
}
```
</CodeGroup>

### Resize Operation (Bilinear Interpolation)

Both implementations use bilinear interpolation but with different access patterns:

- **AOS**: Loads complete pixels at once, good for interpolation
- **SOA**: Must interpolate each channel separately, more overhead

**Result**: AOS typically 1.5-2x faster for resize operations

## When to Use Each Approach

### Choose AOS when:

✓ Implementing algorithms that process complete pixels
✓ Working with color space transformations (RGB → HSV, etc.)
✓ Performing geometric transformations (rotate, scale)
✓ Code simplicity and readability are priorities
✓ Working with smaller images (< 2MB) that fit in L3 cache
✓ Targeting older CPU architectures

### Choose SOA when:

✓ Processing individual color channels independently
✓ Implementing histogram-based algorithms
✓ Performing channel-specific operations (contrast, brightness)
✓ Optimizing for modern CPUs with large SIMD units
✓ Working with very large images (> 10MB)
✓ Memory bandwidth is the bottleneck
✓ Compiler auto-vectorization is important

## Comparison Table

| Aspect | AOS (imtool-aos) | SOA (imtool-soa) |
|--------|------------------|------------------|
| **Memory Layout** | `[RGB][RGB][RGB]...` | `[RRR...][GGG...][BBB...]` |
| **Pixel Access** | ✓ Excellent | ○ Moderate |
| **Channel Access** | ○ Inefficient | ✓ Excellent |
| **Cache Efficiency (full pixel)** | ✓ High | ○ Moderate |
| **Cache Efficiency (channel)** | ✗ Low | ✓ High |
| **Vectorization Potential** | ○ Moderate | ✓ High |
| **Code Complexity** | ✓ Simple | ○ Moderate |
| **Resize Performance** | ✓ Better | ○ Slower |
| **CutFreq Performance** | ○ Good | ✓ Better |
| **Compress Performance** | ✓ Better | ○ Good |
| **Memory Usage** | Same | Same |

## Benchmarking

You can compare performance using the `perf` tool:

```bash
# Benchmark AOS
perf stat -a -e power/energy-pkg/ \
  ./build-release/imtool-aos/imtool-aos input.ppm output.ppm compress

# Benchmark SOA  
perf stat -a -e power/energy-pkg/ \
  ./build-release/imtool-soa/imtool-soa input.ppm output.ppm compress
```

<Info>
Both implementations are validated to produce bit-identical results, so you can confidently choose based on performance alone.
</Info>
