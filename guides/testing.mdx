## Overview

ImgTool includes comprehensive testing with Google Test framework, covering both unit tests for individual components and functional tests for end-to-end validation.

## Test Structure

The testing suite is organized into three categories:

### Unit Tests

- **utest-common**: Tests for common utilities (progargs, binary I/O, info functions)
- **utest-imgaos**: Tests for AOS image implementation
- **utest-img-soa**: Tests for SOA image implementation

### Functional Tests

- **ftest-aos**: End-to-end tests for AOS implementation
- **ftest-soa**: End-to-end tests for SOA implementation

## Running Tests

### Build Tests

Tests are built automatically with the main project:

```bash
cmake -S . -B build-release -DCMAKE_BUILD_TYPE=Release
cmake --build build-release
```

<Note>
  The main CMakeLists.txt enables testing with `enable_testing()` at line 55.
</Note>

### Run All Unit Tests

<CodeGroup>
```bash All Tests with CTest
ctest --test-dir build-release
```

```bash Verbose Output
ctest --test-dir build-release --verbose
```

```bash Individual Test Suite
./build-release/utest-common/utest-common
./build-release/utest-imgaos/utest-imgaos
./build-release/utest-img-soa/utest-img-soa
```
</CodeGroup>

## Functional Tests

### Overview

Functional tests validate end-to-end workflows by comparing output files.

### Test Scripts

Two comparison scripts are provided in both `ftest-aos/` and `ftest-soa/` directories:

#### compare_cppm.sh

Compares compressed CPPM files (source/ftest-aos/compare_cppm.sh):

```bash
#!/bin/bash
if [ "$#" -ne 2 ]; then
    echo "Usage: $0 <image1.cppm> <image2.cppm>"
    exit 1
fi

g++ -o ftest-compress ftest-compress.cpp -std=c++11
./ftest-compress "$1" "$2"
rm -f ftest-compress
```

**Usage**:
```bash
cd build-release/ftest-aos
./compare_cppm.sh output1.cppm output2.cppm
```

<Accordion title="What compare_cppm.sh checks">
  The script compiles and runs ftest-compress.cpp which compares:
  - Magic number (C6)
  - Image dimensions (width, height)
  - Max color value
  - Number of color table entries
  - Color table data (byte-by-byte)
  - Pixel index data (byte-by-byte)
</Accordion>

#### compare_images.sh

Compares PPM image files (source/ftest-aos/compare_images.sh):

```bash
#!/bin/bash
if [ "$#" -ne 2 ]; then
    echo "Usage: $0 <image1.ppm> <image2.ppm>"
    exit 1
fi

g++ -o compare_images tester.cpp -std=c++11
./compare_images "$1" "$2"
rm -f compare_images
```

**Usage**:
```bash
cd build-release/ftest-aos
./compare_images output1.ppm output2.ppm
```

### Functional Test Workflow

<Steps>
  <Step title="Generate test outputs">
    Run ImgTool operations on test images:
    ```bash
    ./build-release/imtool-aos/imtool-aos \
      input/test.ppm \
      output/test-aos.cppm \
      compress
    ```
  </Step>
  
  <Step title="Generate reference outputs">
    Generate expected outputs using a verified implementation.
  </Step>
  
  <Step title="Compare outputs">
    Use comparison scripts:
    ```bash
    cd ftest-aos
    ./compare_cppm.sh ../output/test-aos.cppm ../output/reference.cppm
    ```
  </Step>
  
  <Step title="Validate results">
    Check for "The files are identical" message.
  </Step>
</Steps>

### CPPM File Format Validation

The functional test for compression validates CPPM format (source/ftest-aos/ftest-compress.cpp:10-18):

```cpp
struct CPPMFile {
    std::string magic_number;       // "C6"
    int width;                      // Image width
    int height;                     // Image height
    int max_color_value;            // 255 or 65535
    int num_color_entries;          // Number of unique colors
    std::vector<uint8_t> color_table;   // RGB values
    std::vector<uint8_t> pixel_data;    // Color indices
};
```

<Note>
  CPPM is a custom compressed format that stores unique colors once and uses indices for pixels.
</Note>

## Google Test Framework Usage

### Test Macros

ImgTool tests use standard Google Test macros:

<CodeGroup>
```cpp Basic Tests
// Simple test
TEST(TestSuiteName, TestName) {
    EXPECT_EQ(actual, expected);
    ASSERT_TRUE(condition);
}
```

```cpp Test Fixtures
// Test with setup/teardown
class MyTest : public ::testing::Test {
protected:
    void SetUp() override { /* setup code */ }
    void TearDown() override { /* cleanup code */ }
};

TEST_F(MyTest, TestCase) {
    // test code
}
```

```cpp Assertions
EXPECT_EQ(a, b);     // Continue on failure
ASSERT_EQ(a, b);     // Stop on failure
EXPECT_TRUE(cond);
EXPECT_FALSE(cond);
EXPECT_NEAR(a, b, tolerance);
EXPECT_THROW(statement, exception_type);
```
</CodeGroup>

### Test Main Function

All test files use Google Test's main:

```cpp
int main(int argc, char **argv) {
    ::testing::InitGoogleTest(&argc, argv);
    return RUN_ALL_TESTS();
}
```

## Troubleshooting

<Accordion title="Tests fail to build">
  Ensure Google Test was fetched correctly:
  ```bash
  cmake -S . -B build-release --fresh
  cmake --build build-release
  ```
</Accordion>

<Accordion title="Tests timeout">
  Some tests create temporary files. Ensure sufficient disk space and I/O performance.
</Accordion>

<Accordion title="File not found errors in tests">
  Unit tests create temporary files in the current directory. Run tests from the build directory:
  ```bash
  cd build-release/utest-imgaos
  ./utest-imgaos
  ```
</Accordion>

<Accordion title="Comparison scripts fail">
  Ensure g++ is installed and supports C++11:
  ```bash
  g++ --version
  ```
  
  The scripts compile temporary test executables.
</Accordion>

## Best Practices

1. **Run tests after every build**: Catch regressions early
2. **Use Debug builds for test development**: Better error messages
3. **Add tests for new features**: Maintain test coverage
4. **Test error conditions**: Not just happy paths
5. **Clean up temporary files**: Tests create .ppm and .cppm files
6. **Use descriptive test names**: Make failures easy to understand

<Warning>
  Some tests create temporary files (test_output.cppm, output.ppm, etc.) in the current directory. Clean these up periodically.
</Warning>
